# Продуктовый спек: last-minute тревел-агрегатор

> **Область документа:** только продуктовая часть. Технику (стек, схема БД, API-контракты, архитектура скрапера, TDD) сознательно НЕ описываем — это обсуждается с Claude Code отдельно. Здесь: что строим, зачем, для кого, user stories, фазы и критерии готовности.
>
> **Статус:** черновик v0. Первый источник — Join Up Baltic. Название продукта пока не выбрано.

---

## 1. Одной строкой

Инструмент, который в **последнюю минуту** (окно вылета ~1–3 недели) помогает найти выгодный тур из Риги, показывает **историю цены** («было / стало») и со временем подсказывает, **брать сейчас или подождать**.

---

## 2. Проблема и сценарий

Классический кейс: тебе только что подтвердили отпуск. До возможного вылета неделя-две. Ты **не знаешь, куда лететь**, не хочешь переплачивать и не понимаешь, дешевеет предложение или дорожает.

Существующие сервисы заточены под «планируй заранее»: ты выбираешь страну и даты за месяцы. Под горящку с прозрачностью цены — почти никто.

Наш сервис закрывает ровно этот момент:
- задаёшь **даты, число людей, питание, бюджет** — **без выбора страны**;
- видишь лучшие туры со всех курортов, отсортированные по выгоде;
- по каждому — **как двигалась цена** за время наблюдения;
- получаешь **пинг**, когда появляется подходящее предложение или цена падает ниже твоего порога.

---

## 3. Чем отличается от Aviasales / Skyscanner / Level.Travel

Рынок агрегаторов плотный, поэтому отличие — не «всё в одном месте», а четыре вещи, которых у них нет вместе:

1. **Фокус на последней минуте**, а не на планировании за месяцы.
2. **Поиск без направления** — «неважно куда, вот бюджет и даты, покажи лучшее».
3. **Прозрачность цены** — история движения цены конкретного оффера, а не только цифра «сейчас».
4. **Реальность отзывов** — разрыв между официальными звёздами и настроением живых гостей (защита от «5★, а по факту плохо»).

---

## 4. Продуктовые принципы (ключевые решения)

Это решения, принятые на этапе проектирования. Claude Code должен их держать как рамку.

- **История цены — это ядро, а не фича.** Собираем снимки цен с первого дня. Без накопленной истории продукт не имеет смысла, поэтому сбор данных первичен, витрина — вторична.
- **Поиск без направления — обязательный режим.** Направление (страна/курорт) — это фильтр, а не обязательный вход. Обязательные входы: даты, число людей, питание, бюджет.
- **Уведомления — тоже ядро, не украшение.** В сценарии «уведомили про отпуск» человек не будет мониторить руками. Сохранённый поиск + пинг — то, что заставляет возвращаться. Без этого тул мёртв.
- **Собираем и обычные, и горящие туры.** Обычные нужны, чтобы копить историю: когда оффер попадёт в «горящие», мы уже увидим, реально ли он подешевел или, наоборот, стал дороже своего среднего. Признак «горящий» фиксируется **на момент снимка цены**, а не жёстко на оффере — иначе не поймать «горящий, но дороже».
- **Частота обновления зависит от близости вылета.** Чем ближе дата вылета, тем чаще снимаем цену (движение живёт в последней неделе). Обновлять всё одинаково часто нельзя — источник забанит. Это продуктовое поведение; конкретное расписание — в тех. части.
- **Отзывы — от реальных гостей, из нескольких источников.** Ценность не в самом рейтинге, а в **разрыве** между звёздами оператора и настроением гостей.
- **«Бери / жди» — зарабатывается данными, не декларируется.** Гипотеза «мало осталось + скоро вылет = дешевле» на старте не доказана и может быть перевёрнута. Поэтому историю показываем сразу, а рекомендацию включаем позже, когда накопленные данные её подтвердят.
- **Начинаем с одного источника (Join Up Baltic) и одного аэропорта (Рига).** Доводим пайплайн до ума на нём, потом клонируем на других операторов. Источники — сменные модули.

## 5. Доменная модель (концептуально)

> Это продуктовый словарь — что за сущности существуют в мире продукта и какие правила между ними. Поля, типы и связи БД — на тех. сессию с Claude Code.

- **Оператор** — источник данных (Join Up, дальше Novatours и т.д.). Сменный модуль.
- **Отель** — единая карточка отеля с приведённым названием, страной, курортом, официальными звёздами и координатами. Якорь для привязки отзывов.
- **Оффер** — конкретное предложение. **Правило идентичности оффера** (что делает его уникальным): отель + дата вылета + число ночей + питание + тип номера + состав туристов + аэропорт вылета. Именно эта связка отслеживается во времени.
- **Снимок цены** — точка в истории оффера: когда сняли, цена, **остаток мест** («осталось 3» живёт именно здесь), был ли оффер «горящим» в этот момент. Самая объёмная сущность.
- **Источник отзывов** — по отелю × платформе: рейтинг, число отзывов, оценка настроения, краткое саммари.
- **Подписка (watchlist)** — сохранённый поиск: набор фильтров (направления, даты, ночи, питание, состав, порог цены) + канал уведомления.
- **Срабатывание (alert)** — факт уведомления: по какой подписке, какой оффер, причина (новое совпадение / падение цены / ниже порога).
- **Пользователь** — на старте это ты; заглушка нужна ради привязки уведомлений (напр. Telegram).

## 6. User Stories

Роль одна — **отдыхающий** (человек, ищущий горящий тур). Ниже сгруппировано по функциям, с критериями приёмки. В скобках — фаза.

### Поиск и просмотр (v0)
- **Как отдыхающий, я хочу задать даты, число людей и питание без выбора страны**, чтобы увидеть все подходящие туры со всех курортов, отсортированные по цене.
  - *Готово, когда:* поиск работает без обязательного направления; выдача отсортирована по цене; охватывает все доступные курорты источника.
- **Как отдыхающий, я хочу фильтровать выдачу** по бюджету, длительности, питанию и курорту, чтобы сузить результаты.
- **Как отдыхающий, я хочу видеть по каждому туру** отель, даты, питание, аэропорт вылета и текущую цену, чтобы понять суть предложения.
- **Как отдыхающий, я хочу видеть пометку «горящий»**, чтобы отличать спец-предложения.

### История цены (v1)
- **Как отдыхающий, я хочу видеть график изменения цены тура** за время наблюдения, чтобы понять тренд.
  - *Готово, когда:* по офферу отображается история цены за весь период наблюдения.
- **Как отдыхающий, я хочу видеть предупреждение «горящий, но дороже своего среднего»**, чтобы не купиться на ложную выгоду.
- **Как отдыхающий, я хочу видеть остаток мест** по предложению, чтобы оценить дефицит.

### Подписки и уведомления (v2)
- **Как отдыхающий, я хочу сохранить свой поиск как подписку** (даты, бюджет, питание, состав, порог цены), чтобы не мониторить руками.
- **Как отдыхающий, я хочу получать уведомление в Telegram**, когда появляется подходящий тур или цена падает ниже моего порога, чтобы успеть в последнюю минуту.
- **Как отдыхающий, я хочу управлять подписками** (список, включить/выключить, удалить), чтобы не получать лишнего.

### Отзывы (v3)
- **Как отдыхающий, я хочу видеть реальный рейтинг отеля от гостей** (не только звёзды оператора), чтобы не нарваться на «5★ с плохими отзывами».
- **Как отдыхающий, я хочу видеть разрыв между официальными звёздами и реальным настроением гостей**, чтобы отсеять переоценённые отели.
- **Как отдыхающий, я хочу краткое саммари отзывов**, чтобы быстро понять плюсы и минусы отеля.

### Рекомендация «бери / жди» (v4)
- **Как отдыхающий, я хочу подсказку «бери сейчас / можно подождать» с обоснованием**, чтобы принять решение без риска переплатить или упустить.
  - *Готово, когда:* подсказка показывается только там, где накопленных данных достаточно, и сопровождается объяснением, на чём основана.

## 7. Роадмап по фазам

Каждая фаза — самостоятельно полезный кусок. Не начинаем следующую, пока текущая не «готова».

### v0 — Каталог и поиск (локально, дни)
Цель: собрать офферы Join Up (обычные + горящие, вылет из Риги) и дать поиск без направления.
- [ ] источник тянет обычные и горящие туры Join Up с вылетом из Риги
- [ ] офферы сохраняются с уникальной идентичностью (см. правило идентичности оффера)
- [ ] работает поиск без выбора направления (даты + состав + питание + бюджет → выдача по всем курортам, сортировка по цене)
- [ ] фильтры: бюджет, длительность, питание, курорт
- [ ] всё крутится локально на твоей машине

### v1 — История цены (автосбор)
Цель: начать копить историю и рисовать её.
- [ ] офферы опрашиваются автоматически, чаще — по мере приближения вылета
- [ ] по каждому офферу есть график цены за период наблюдения
- [ ] считается и показывается флаг «горящий, но дороже своего среднего»
- [ ] остаток мест сохраняется в каждом снимке
- [ ] (опционально) переезд сбора на постоянно работающий хост

### v2 — Подписки + Telegram
Цель: возвращаемость — сервис пингует сам.
- [ ] можно сохранить поиск как подписку с параметрами и порогом цены
- [ ] Telegram-бот шлёт уведомление при новом совпадении или падении ниже порога
- [ ] подписками можно управлять (список, вкл/выкл, удалить)

### v3 — Отзывы
Цель: разрыв «звёзды vs реальность».
- [ ] отель из оффера привязывается к своей карточке минимум на одном внешнем источнике отзывов
- [ ] показывается реальный рейтинг + число отзывов + краткое саммари
- [ ] показывается разрыв «официальные звёзды vs настроение гостей»
- [ ] где возможно — отзывы из нескольких источников

### v4 — Бери / жди
Цель: рекомендация на данных.
- [ ] накоплены полные жизненные циклы достаточного числа офферов
- [ ] гипотеза «остаток ↔ цена» проверена эмпирически (подтверждена или отвергнута)
- [ ] подсказка «бери / жди» показывается с обоснованием, только где данные позволяют

## 8. Вне скоупа (чего НЕ делаем)

- Не принимаем оплату и не оформляем брони — только редирект на оператора.
- Не являемся турагентством: без юр. ответственности перед клиентом, без обязательств Package Travel Directive, без страховки от неплатёжеспособности.
- Не планировщик «на будущее» — фокус строго на последней минуте.
- Не мультигород и не сложные составные маршруты.
- Пока только вылет из Риги.
- Пока только один оператор (Join Up) — остальные позже.

## 9. Риски и допущения

- **Главный риск — доступ к данным.** Всё зависит от того, отдаёт ли источник данные стабильно. Если оператор начнёт блокировать сбор — продукт встаёт. Это риск №1, важнее любого кода. Смягчение: вежливый неравномерный опрос, один источник на старте, аккуратность с частотой.
- **Гипотеза может быть перевёрнута.** «Мало осталось + скоро вылет = дешевле» — не факт; дефицит часто толкает цену **вверх**. Поэтому «бери / жди» не декларируем на старте, а проверяем на собранных данных (v4).
- **Привязка отеля к отзывам — сложная.** Транслит и разные названия («Sokol Resort» vs «Hotel Sokol Beach Crete») требуют нечёткого сопоставления. Отдельная возня — поэтому v3, не раньше.
- **Юр. рамка сейчас мягкая, но не навсегда.** Для личного использования — ок. Коммерциализация позже вскроет условия использования источников и вопрос перераспространения данных. Не блокирует сейчас, но заложи в голову на будущее.
