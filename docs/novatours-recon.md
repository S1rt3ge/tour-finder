# Разведка источника #2: Novatours (novatours.lv)

> Дата: 2026-07-18. Метод: анализ фронтенда + перехват сети во встроенном браузере
> + прямые пробы эндпоинта. Цель — оценить, можно ли добавить Novatours вторым
> оператором ради расширения списка стран (Join Up даёт потолок 8 из Риги).

## Вердикт

**Добавить можно, выгода большая, но это существенно бо́льшая работа, чем Join Up** —
из-за архитектуры и защиты доступа. Не «быстрый плюс оператор», а отдельный
мини-проект уровня всей интеграции Join Up.

## Архитектура

- **Magento 2 + ScandiPWA** (React PWA-тема для Magento) + **GraphQL**.
  Подтверждено: `/static/frontend/scandipwa/theme/...`, эндпоинт `/lv/graphql`,
  Apollo APQ (persisted queries: `graphql?hash=2832062463&identifiers_1=[...]`).
- **Общий бэкенд по Балтии.** Материнский бренд — **Novaturas** (`novaturas.lt`);
  медиа и PIM общие. LV/LT/EE-сайты, вероятно, ходят в один backend.
- **Каталог отелей — PIM:** `https://pim.novatours.eu/Hotels/{id}/...`, id числовые
  (12201, 24172…) + коды с зашитой страной: `VNSMUINEBO` (Вьетнам, Муйне),
  `GRHDANAOS` / `GRXKASAPAK` (Греция). Страну можно доставать из кода отеля.

## Выгода по странам (главное ради чего)

Карусель и PIM показывают **дальнемагистральные** направления, которых у Join Up
из Риги нет: **Вьетнам, Шри-Ланка** и т.д. Один Novatours кратно расширяет
географию сверх 8 стран Join Up.

## Сложность доступа (ключевое отличие от Join Up)

- Join Up: открытый REST JSON, бьёшь curl'ом — и всё.
- Novatours: **прямой `POST /lv/graphql` curl'ом → 302** (эдж/WAF заворачивает
  запрос без браузерной сессии). GET APQ-урла отдаёт HTML-оболочку, не JSON.
  То есть API завязан на сессию/куки/заголовки (Store, возможно подпись APQ).
  Значит headless-доступ сложнее: нужна эмуляция сессии либо игра с APQ-хешами.
- Встроенный браузер поиск-виджет не догидрировал (поля пустые), живой
  tour-search GraphQL-запрос за эту сессию поймать не удалось.

## Что нужно, чтобы построить модуль (оценка)

1. **Поймать сам tour-search GraphQL-запрос** (документ + переменные: направление,
   даты, ночи, пакс, флаг last-minute, пагинация) и форму ответа. Чище всего —
   пользователь в своём Chrome: DevTools → Network → фильтр `graphql` → сделать
   поиск тура → правой кнопкой по запросу → **Copy as cURL** → отдать мне. Тогда
   декодирую полностью и не воюю с WAF.
2. **GraphQL-клиент-источник** (`sources/novatours.py`) под тот же интерфейс, что
   Join Up (`destinations`/`search_pages`/`normalize`). Клиент другой (GraphQL+сессия,
   не голый REST).
3. **Маппинг модели Magento-travel → наша схема** (идентичность оффера, снимок цены).
4. **Матчинг отелей между операторами** — та самая «сложная привязка» из спека
   (SPEC §9). На старте можно показывать отдельными карточками; схлопывание
   «тот же отель у двух операторов» — отдельная возня.

## Рекомендация

Novatours стоит добавления — ради дальних стран. Но сначала нужен **чистый
перехват search-запроса**. Самый дешёвый путь: пользователь экспортит его как
cURL из своего браузера (WAF там пройден живой сессией), я декодирую и строю
модуль. Альтернатива — глубже ковырять headless-сессию/APQ, дольше и с риском
упереться в бот-защиту.

## Проверенные факты (для следующей сессии)

- Эндпоинт: `https://www.novatours.lv/lv/graphql` (Store-заголовок `lv`).
- Прямой POST без сессии → 302. APQ persisted queries по `hash` + `identifiers`.
- Отели: `pim.novatours.eu/Hotels/{id}`, коды стран-префиксом (VN, GR, …).
- Стек: Magento 2 + ScandiPWA + Apollo GraphQL. Виджеты рассылки (soundest/klaviyo),
  чат beesender — шум, не наше.
