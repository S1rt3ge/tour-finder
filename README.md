# tour-finder

Last-minute агрегатор туров из Риги. Продукт: [SPEC.md](SPEC.md).
Источник данных: Join Up Baltic через неофициальный JSON API — [docs/joinup-api-recon.md](docs/joinup-api-recon.md).

## Прод (облако, $0/мес)

- **Витрина:** https://tour-finder-rose.vercel.app (Vercel, деплой `vercel deploy --prod`)
- **Сборщик:** GitHub Actions [`collect.yml`](.github/workflows/collect.yml), cron каждые 2 часа; сам решает, какие ярусы пора обновить
- **БД:** Supabase Postgres (transaction pooler, порт 6543). Единственный секрет — `DATABASE_URL` (GitHub Secrets + Vercel env)
- Один код на оба бэкенда: `DATABASE_URL` не задан → локальный SQLite `data/tourfinder.db`
- Перенос локальной истории в облако: `scripts/migrate_sqlite_to_pg.py --wipe` (для миграции бери session-порт 5432)

## Запуск локально (Windows)

```powershell
py -3.12 -m venv .venv
.venv\Scripts\pip install -e .

# сбор данных (обычные + горящие туры, вылет из Риги, окно 30 дней)
.venv\Scripts\python -m tourfinder.cli fetch

# веб-интерфейс на http://127.0.0.1:8000
.venv\Scripts\python -m tourfinder.cli serve
```

`fetch` можно ограничить для проверки: `fetch --destinations c_8 --max-pages 2 --days 14`.
Числа по базе: `python -m tourfinder.cli stats`.

### Состав тургруппы

Цена зависит от состава, поэтому поиск отдаёт только тот состав, который реально
собран. Составы задаются через `--pax` (повторяемый):

```powershell
python -m tourfinder.cli fetch --pax "2"          # 2 взрослых (по умолчанию)
python -m tourfinder.cli fetch --pax "2+1:7"      # 2 взрослых + 1 ребёнок 7 лет
python -m tourfinder.cli fetch --pax "2+2:6,8"    # 2 взрослых + дети 6 и 8 лет
```

`collect` по умолчанию собирает `["2", "2+1:7"]`. В UI, если по выбранному составу
данных нет, показывается что собрано и какой командой добрать нужный.

БД — SQLite в `data/tourfinder.db` (в git не попадает). Каждый запуск `fetch`
пишет снимок цены по каждому найденному офферу — история копится с первого дня.

## Автосбор (v1)

`collect` — точка входа для планировщика. Гоняет ярусы по близости вылета,
пропуская те, что ещё свежие, и не даёт двум сборам наслоиться:

| ярус | вылет через | как часто снимаем |
|------|-------------|-------------------|
| near | 1–7 дней    | каждые 4 ч        |
| mid  | 8–14 дней   | каждые 12 ч       |
| far  | 15–45 дней  | раз в сутки       |

В проде это делает GitHub Actions. Локальная альтернатива (если облако не нужно) —
Планировщик Windows: `.\scripts\register-scheduler.ps1` (снять —
`unregister-scheduler.ps1`); задача крутит `collect` под `pythonw`, лог `data\collect.log`.

## Подписки и уведомления (v2)

Вместо Telegram — уведомления прямо во вкладке браузера со звуком.

1. Задай поиск (даты, бюджет, питание, страна) и нажми **«Сохранить как подписку»**.
   Бюджет в поиске = порог цены подписки.
2. Нажми **«🔔 включить звук»** (браузер требует клик, чтобы разрешить звук).
3. Вкладка раз в минуту опрашивает базу. Появился новый подходящий тур или
   цена упала ниже прошлого срабатывания — звук + всплывашка снизу справа.

Совпадения копятся и при закрытой вкладке: `collect` прогоняет подписки в конце
каждого сбора, при следующем открытии вкладки увидишь непрочитанные.
Подписками можно управлять там же: вкл/выкл, удалить.

## Отзывы гостей (v3)

Join Up рейтинги не отдаёт, поэтому отзывы тянем из внешней площадки, сопоставляя
отель по названию + гео. Источники сменные (интерфейс `sources/reviews.py`).
Референс-провайдер — Google Places; нужен свой ключ (free tier хватает):

```powershell
$env:GOOGLE_PLACES_API_KEY = "ваш-ключ"
.venv\Scripts\python -m tourfinder.cli reviews --limit 50   # обогатить 50 отелей
```

Без ключа команда ничего не делает. В выдаче у отеля появляется рейтинг гостей и
бейдж **разрыва со звёздами**: «хуже звёзд» (гости оценивают ниже класса отеля,
разрыв ≤ −0.75) или «лучше звёзд» (≥ +0.75). Порог `--max-age-days` — как часто
перезапрашивать. Ключ отеля к площадке кэшируется, повторные запуски дёшевы.
